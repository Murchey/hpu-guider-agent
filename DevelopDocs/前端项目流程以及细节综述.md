# 文途智行 (Wentu Zhixing) 项目流程以及细节综述

## 1. 项目背景与核心定位

“文途智行”是一款深度融合了 Vue 3 前端框架与大语言模型（LLM）能力的智慧导游助手。在旅游行业数字化转型的背景下，本项目致力于解决传统导览工具“信息碎片化”、“交互单向性”以及“缺乏个性化”的痛点。

### 1.1 核心价值主张
- **保姆级个性化规划**：通过首页的用户画像问卷（涵盖旅行人数、天数、预算、偏好风格等），系统能精准捕捉用户的隐性需求，从而生成极具针对性的行程方案。
- **深度文化交互**：不同于简单的景点介绍，AI 能够模拟专业导游的口吻，结合历史背景、趣闻轶事进行深度解说。
- **结构化交互闭环**：利用自定义的 JSON 标签协议（`[SCENE_DATA]`, `[SELECT_DATA]`），将 AI 的生成能力转化为前端可操作的 UI 组件，极大地提升了用户体验的确定性。
- **多模态感知能力**：支持图片上传与分析，用户在景区现场拍摄照片即可获得即时的解说与指引。

## 2. 技术栈架构设计与选型逻辑

项目采用了行业领先的前端技术栈，旨在构建一个高性能、易维护且具备高度交互性的单页应用（SPA）。

### 2.1 核心框架层
- **Vue 3 (Composition API)**：采用组合式 API 架构，利用 `setup` 语法糖和 `ref/computed` 等响应式工具，将复杂的对话逻辑、图片处理逻辑与 UI 渲染进行解耦。
- **TypeScript**：作为开发语言，定义了严谨的接口类型（如 `Message`, `ApiSettings`, `UserProfile` 等），在编译阶段拦截了 90% 以上的常见错误。
- **Vite 5**：作为构建引擎，其极速的开发服务器启动和模块热更新（HMR）特性，极大地提升了开发效率。

### 2.2 UI 表现层
- **Element Plus**：选用了成熟的组件库作为 UI 基础，并对其进行了深度的样式定制（如对话气泡、加载动画、响应式抽屉等）。
- **Markdown-it**：AI 的输出往往包含复杂的 Markdown 语法（加粗、列表、代码块、引用等）。我们配置了 `markdown-it` 渲染引擎，并添加了 `linkify` 和 `breaks` 插件，确保输出内容的美观与易读。
- **视觉特效系统**：
    - **视差滚动 (Parallax)**：通过响应式监听鼠标坐标，动态计算背景图层的 `translate` 偏移，增强页面的纵深感。
    - **碎片飞散 (Shatter)**：利用 DOM 碎片网格化技术，模拟图片在交互时的物理破碎效果。

### 2.3 网络与协议层
- **Axios**：负责标准的 RESTful 请求，如 Coze 平台的文件上传接口。
- **Fetch API & SSE (Server-Sent Events)**：对话功能的核心采用了流式传输协议。相比传统的 AJAX 请求，SSE 能够让服务器持续向客户端推送数据片段，从而实现“打字机”式的实时展示效果。

## 3. 核心功能模块深度解析

### 3.1 首页 (IndexPage.vue)：从视觉到画像
首页不仅是一个欢迎界面，更是用户数据的入口。
- **用户画像问卷逻辑**：
  - 设计了一个包含 10 余个维度的问卷（`el-form`），包括出发日期、交通方式、出行风格（松弛度假、高效打卡、人文深度、自然户外、美食寻味、城市漫游）。
  - 数据持久化：表单数据通过 `JSON.stringify` 存入 `localStorage` 的 `user-profile` 字段，确保用户刷新页面后数据不丢失。
- **动态视觉交互**：首页背景采用了多层级视差设计，配合 `shatter-sensor` 捕捉用户的感官交互，营造出一种“智慧旅游”的科技氛围。

### 3.2 对话中心 (AiDialoguePage.vue)：智慧交互的核心
这是整个项目逻辑最密集的模块，涉及多维度的状态管理。
- **多模态输入流**：
  - **图片上传**：系统根据当前配置的 AI 提供商自动切换上传模式。如果是 Coze，则先通过 `/v1/files/upload` 接口获取 `file_id`；如果是硅基流动，则直接转为 Base64。
  - **工作模式切换**：通过 `el-segmented` 提供的分段控制器，用户可以在“互动问答”与“景点介绍”模式间快速切换，系统会自动在 Prompt 中注入对应的 `task_type` 标识。
- **消息渲染引擎**：
  - 每条消息都经过 `formatAssistantMessage` 处理，不仅处理 Markdown，还会识别并提取自定义交互协议中的 JSON 标签。
  - 使用 `nextTick` 和 `scrollToBottom` 确保在消息流更新时，页面始终自动滚动到最新位置。

### 3.3 地图规划 (MapPlanningPage.vue)：可视化行程管理
将 AI 生成的抽象文本转化为具象的地理路线，是“文途智行”闭环体验的重要一环。
- **高德地图 SDK 集成**：利用 `@amap/amap-jsapi-loader` 动态加载高德地图 2.0 及其核心插件（Driving、Walking、Transit 等）。
- **跨城与复杂路线规划**：
  - 通过 `axios` 结合高德 Web 服务 API（`direction/transit/integrated`）实现跨城火车、高铁的路线数据获取。
  - 对于缺失轨迹的火车路线，采用数学插值算法生成“模拟弧线（Arc Path）”，提升视觉体验。
  - 支持多交通方式混合渲染，不同模式对应不同颜色的轨迹与样式。
- **交互式行程调整**：
  - **拖拽排序**：借助 `vue-draggable-plus`，用户可以在侧边栏直观地拖拽景点调整游览顺序，地图会自动重绘规划路线。
  - **模式切换**：用户可为行程中的每一段单独指定出行方式（驾车、公交、步行、骑行）。
  - **APP 唤起**：通过组装 `amapuri://` 协议链接，支持一键唤起手机端的高德地图 APP 进行实地导航。

### 3.4 设置中心 (SettingPage.vue)：引擎配置与提示词工程
- **API 动态热切换**：支持用户自定义 API Key、Base URL 和模型 ID。这种设计使得应用不依赖于单一服务商，具备极高的灵活性。
- **系统提示词 (System Prompt) 注入**：这是控制 AI 行为的“大脑”。Prompt 中详细定义了 AI 的身份（文途智行）、工作阶段（读取画像 -> 意图确认 -> 景点筛选 -> 规划生成 -> 深度讲解）以及输出规范。

## 4. 自定义交互协议 (Custom Interaction Protocol) 的实现原理

为了打破大模型只能输出“纯文本”的局限，本项目独创了一套基于标签的 UI 触发协议。

### 4.1 协议定义
- **单选卡 `[SCENE_DATA]`**：
  - 格式：`[SCENE_DATA]{"description": "...", "btn1": "...", ...}[/SCENE_DATA]`
  - 前端逻辑：使用正则表达式 `\[SCENE_DATA\]([\s\S]*?)\[\/SCENE_DATA\]` 全局匹配。匹配成功后，解析中间的 JSON 字符串，将其属性绑定到 `sceneButtons` 响应式对象，并触发 `sceneDialogVisible = true`。
- **多选卡 `[SELECT_DATA]`**：
  - 格式：`[SELECT_DATA]{"description": "...", "options": [...]}[/SELECT_DATA]`
  - 前端逻辑：类似于单选卡，但解析出的 `options` 会被渲染为 `el-checkbox-group`。

### 4.2 交互闭环 (`[USER_CHOICE]`)
当用户在弹窗中做出选择后，系统不会在聊天框中显示一条生硬的用户回复，而是通过 `handleSendHidden` 函数发送一条“隐藏指令”：
1. **构建 Payload**：`[USER_CHOICE]{"value": "..."}[/USER_CHOICE]`。
2. **静默发送**：将该指令追加到 `messages` 数组中，但在模板渲染时根据标识隐藏，确保对话流的简洁。
3. **AI 响应**：AI 看到该标签后，识别用户意图，直接推进到业务流程的下一阶段。

## 5. 关键技术细节与原理分析

### 5.1 SSE 流式输出的异常处理与缓冲区管理
在处理流式数据时，最常见的挑战是“JSON 截断”。由于网络波动，一个完整的 `data: {...}` 可能会被拆分成多次推送。
- **解决方案**：在读取流时，使用一个临时字符串变量 `buffer` 存储未处理的数据片段。
- **解析逻辑**：
  ```typescript
  const lines = (buffer + chunk).split('\n');
  buffer = lines.pop() || ''; // 最后一行可能不完整，留到下次处理
  for (const line of lines) {
    if (line.startsWith('data: ')) {
      // 处理完整的 JSON 片段
    }
  }
  ```

### 5.2 移动端响应式布局的细节处理
为了适配从 iPhone SE 到 4K 显示器的各种设备：
- **动态计算属性**：使用 `computed` 根据 `isMobile` 状态动态返回弹窗宽度（`dialogWidth`）和字体大小。
- **输入框交互优化**：在移动端，通过 `autosize` 限制输入框行数，并使用 `append-to-body` 确保弹窗不被键盘遮挡。

### 5.3 提示词工程 (Prompt Engineering) 的应用
项目在 `SettingPage.vue` 中内置了长达千字的 System Prompt。其核心技巧包括：
- **少样本提示 (Few-shot)**：在 Prompt 中给出具体的标签调用示例，引导 AI 生成符合规范的 JSON。
- **思维链 (CoT)**：明确要求 AI 按照“读取画像 -> 分析需求 -> 提出建议”的步骤思考，保证逻辑的一致性。
- **角色约束**：严格限定 AI 的称呼为“文途智行”，语气必须亲切且专业。

## 6. 项目业务流程全景梳理

文途智行的业务流程被设计为一个高度自动化的“漏斗式”交互过程，从模糊的意图收集逐步收窄至精确的方案产出。

### 6.1 阶段一：画像感知与初始化 (Sense & Init)
1. **视觉吸引与入口**：用户在首页通过 `IndexPage.vue` 的视差滚动和飞散特效建立初步品牌认知，点击“让AI懂你喜好”开启服务。
2. **多维画像构建**：用户在 `el-drawer` 抽屉中填写包含旅行天数、预算、出发地、风格（如“高效打卡”）、核心关注点等 10 余个维度的问卷。
3. **数据持久化与触发**：点击确认后，表单数据存入 `localStorage`。系统自动跳转至 `AiDialoguePage.vue`，并静默发送一条前缀为 `model:用户画像` 的结构化指令。
4. **AI 预热**：AI 接收到画像后，内部逻辑进入“阶段一：角色激活”，并准备生成个性化开场白。

### 6.2 阶段二：意图确认与场景引导 (Intention Confirmation)
1. **个性化欢迎**：AI 根据画像（如“家庭出游”、“深度人文”）回复一段亲切的欢迎语。
2. **触发单选卡 (Scene Card)**：AI 在回复末尾附带 `[SCENE_DATA]` 标签。前端解析后弹出对话框，提供如“定制今日路线”、“探索历史典故”、“寻找周边美食”等选项。
3. **静默反馈 (Hidden Choice)**：用户点击按钮后，前端通过 `handleSendHidden` 函数发送 `[USER_CHOICE]{"value": "..."}`。此过程对用户透明，保持对话界面的整洁。

### 6.3 阶段三：景点筛选与互动决策 (Selection & Decision)
1. **需求细化**：AI 根据用户选择的场景，进一步推荐 5-8 个具体景点或体验项目。
2. **触发多选卡 (Select Card)**：AI 发送包含 `[SELECT_DATA]` 的 JSON 标签。前端渲染出一个带滚动条的多选列表，允许用户勾选感兴趣的地点。
3. **方案聚合**：用户勾选（如：龙门石窟、白马寺）并确认。前端再次以 `[USER_CHOICE]{"values": [...]}` 形式回传给 AI。

### 6.4 阶段四：全案生成与多模态陪同 (Plan Generation & Multi-modal)
1. **综合方案产出**：AI 结合用户选定的景点、出发地及交通方式，生成包含“最优游览顺序 + 时间表 + 交通指南 + 避坑提示”的 Markdown 方案。
2. **多模态增强**：用户在游览过程中可随时上传现场照片。AI 识别图片内容后，自动触发“现场讲解”逻辑，提供实时的、基于视觉的解说词。
3. **模式动态切换**：用户可随时通过“工作模式”切换器，将 AI 调整为“景点介绍”模式（侧重深度文化）或“互动问答”模式（侧重即时咨询）。

### 6.5 异常与流程恢复机制
- **状态丢失处理**：若用户刷新页面，系统会自动调用 `loadSettings` 从 `localStorage` 恢复最新的 API 配置和用户偏好，确保对话可以无缝继续。
- **协议容错**：若 AI 输出的 JSON 格式有误，前端解析函数会捕获异常并记录日志，同时保留原始文本显示，防止程序崩溃。

## 7. 项目关键代码函数解析

### 7.1 `AiDialoguePage.vue`：AI 对话核心逻辑
- **`sendMessage(text, showUserMessage)`**: 核心发送函数。负责初始化 `AbortController`、构造消息对象、发起流式请求，并在此处拦截、提取并移除 `[SCENE_DATA]` 等协议标签。
- **`formatAssistantMessage(text)`**: 消息渲染函数。调用 `markdown-it` 将 AI 输出的 Markdown 源码转化为 HTML，确保样式符合规范。
- **`handleFileChange(file)`**: 图片上传入口。根据 `imageMode`（Coze 或 Base64）自动切换图片预处理逻辑。
- **`uploadImageToCoze(file)`**: Coze 专有上传函数。将图片通过 `FormData` 异步上传至 Coze 文件服务器，获取 `file_id` 供后续对话使用。
- **`handleSendHidden(text)`**: 隐藏消息发送器。用于发送 `[USER_CHOICE]` 等协议反馈，确保交互闭环且不干扰用户视觉流。
- **对话历史与 Pinia 状态管理**: 引入 `pinia` 并在 `chatStore.ts` 中实现对话持久化。
    - **`createNewChat()`**: 清空当前视图并创建一个带有唯一 ID 的新会话。
    - **`switchChat(id)`**: 切换当前活跃的对话。
    - **`deleteChat(id)`**: 从本地存储中移除指定对话。
    - **`renameChat(id, newTitle)`**: 修改对话会话的标题。
    - **`deleteMessage(index)`**: 删除当前会话中的特定消息。
    - **`editMessageAndTruncate(index, content)`**: 修改用户消息并截断其后的所有 AI 回复，为重新生成对话做准备。
- **消息交互功能**:
    - **复制**: 使用剪贴板 API 实现一键复制消息原文。
    - **修改**: 允许用户对已发送的消息进行二次编辑，提交后系统会自动触发重新发送逻辑并覆盖旧有的后续对话。
    - **删除**: 允许用户精简对话流，移除无关或错误的消息。
- **LocalStorage 持久化**: 自动将 `sessions` 数组序列化存储在 `ai-chat-sessions` 键名下，确保页面刷新后对话记录不丢失。

### 7.2 `SettingPage.vue`：配置与提示词管理
- **`applyCozeSettings()`**: 一键应用函数。从 `MANUAL_API_SETTINGS` 常量中读取预设的 Coze 配置并保存至 `localStorage`。
- **`saveCustomSettings()`**: 自定义保存逻辑。校验用户输入的 Base URL、API Key 等信息，实现配置的持久化。
- **`sendInitialPrompt()`**: 系统引导函数。将 `systemPrompt` 响应式变量的内容发送给 AI，并自动跳转至对话选项卡，完成 AI 角色的“灵魂注入”。
- **`invokeSimulatedAiResponse()`**: **测试专用函数**。允许测试人员手动输入符合协议格式的消息（如地图数据、单选/多选卡片），直接推送到当前对话中，用于验证前端解析和渲染逻辑的准确性。

### 7.3 `MapPlanningPage.vue`：地图规划与路线渲染逻辑
- **`initMap()`**: 核心初始化入口。检测 `.env` 配置的 `VITE_AMAP_KEY` 与安全密钥，调用 `AMapLoader.load` 实例化地图与各类出行方式插件（如 `Driving`、`Walking`、`Transit`）。
- **`updateMap()`**: 防抖封装的地图刷新控制流。在数据加载或发生变更（拖拽排序、改变出行模式）时，清除旧的标记和线路，重新循环遍历 `itineraryList` 并为行程的每一个节点添加对应的标记点。
- **`drawSegments()`**: 分段渲染总控制器。负责调用 `performSearch` 逐步发起两个途经点之间的路径搜索请求。
- **`performSearch(currentMode, isFallback)`**: 递归的路线探测函数。针对 `transit` 模式调用 Web API 请求综合交通方案（跨城火车、市内公交地铁）；针对其他常规模式调用对应的高德 SDK 插件；若规划失败会自动尝试降级（如公交 -> 驾车 -> 绘制直线），确保地图轨迹视觉连贯。
- **`getArcPath(start, end, steps)`**: 插值数学函数。用于弥补由于火车 API 未提供具体地理轨迹导致的地图空白，通过正弦函数计算出带有微凸弧度（Arc Path）的模拟高铁/航线。
- **`openAmapApp()`**: 移动端唤起模块。使用 `amapuri://route/plan/` 协议并结合 `iframe` 进行静默尝试，如果在 1 秒内未成功跳至原生高德 APP，则作为兜底自动在新标签页打开高德地图 Web 版导航。

### 7.4 `IndexPage.vue`：视觉特效与画像收集
- **`handleShatterMouseMove(e)`**: 碎片联动函数。通过计算鼠标位置与 `shatter-sensor` 的相对坐标，动态改变碎片网格中每个单元格的 `transform` 属性，实现物理破碎效果。
- **`getFragmentStyle(row, col)`**: 样式生成器。为每个图片碎片计算背景偏移（`background-position`）和初始位移。
- **`handleProfileSubmit()`**: 画像提交函数（逻辑内嵌于确认按钮）。将 `form` 对象序列化后存入本地，并触发全局跳转信号。

### 7.5 `App.vue`：全局状态与主题控制
- **`handleThemeChange(val)`**: 主题切换函数。控制 `html` 标签的 `.dark` 类名切换，并同步更新 `localStorage` 中的主题偏好。
- **`handleNavigate(tabName)`**: 路由导航模拟。通过修改 `activeName` 实现首页、对话页、设置页之间的平滑切换。
- **`visibilitychange` 监听器**: 状态保活逻辑。在页面切入后台或息屏时，确保关键连接和状态能够被正确处理。（目前仍然无法很好处理切屏、息屏等场景的问题）
