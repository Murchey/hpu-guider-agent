# AI对话气泡检测机制
检测到AI对话气泡底部含有[MAP_DATA]{JSON格式数据}[/MAP_DATA]时，给AI气泡底部增加一个“转到地图”按钮，点击后跳转到地图规划页面。地图规划页面会根据JSON格式数据绘制出地图，并在地图上显示出用户的位置。

# 地图规划交互设计
## json数据格式
{
  "itinerary": [
    {
      "id": "start_01",
      "name": "洛阳龙门站",
      "type": "departure",
      "coordinates": {"lng": 112.463, "lat": 34.593},
      "suggested_duration": 0, 
      "travel_mode": "driving", // 默认出行方式: driving(驾车)/walking(步行)/transit(公共交通)/cycling(骑行)
      "description": "行程起始点"
    },
    {
      "id": "spot_01",
      "name": "故宫博物院",
      "type": "attraction",
      "coordinates": {"lng": 116.397, "lat": 39.918},
      "suggested_duration": 180, 
      "travel_mode": "walking",
      "description": "明清两代皇家宫殿"
    }
    // ... 其他景点/餐厅
  ]
}

前端根据经纬度在地图组件上绘制地图。要求 `itinerary` 数组的第一个元素必须是 `type: "departure"`。`travel_mode` 决定从**当前地点到下一个地点**的出行方式。

## 交互设计
“拖拽重算”的实现逻辑（重点）
1. **固定起始点**：用户无法将其他地点拖动到 "出发地" 之前，确保行程始终从出发地开始。
2. **出行方式选择**：用户可以手动更改每一段行程的出行方式（步行、公交、驾车、骑行），前端即时重绘路径。
3. **初始加载**： 用户输入需求，请求 Coze。Coze 返回包含出发地及景点的 JSON。
3. **前端渲染**： Vue3 拿到 JSON，左侧用 vue-draggable-plus 渲染，右侧在高德地图上打点画线。

用户拖拽： 用户觉得先去景点 C 再去景点 B 更好，将 C 的卡片拖到了 B 前面。

前端本地重算（无缝体验）： * vue-draggable-plus 瞬间改变了数组顺序。

Vue3 监听到数组变化，立刻提取新的经纬度顺序。

前端直接发送请求给高德地图 API：“请告诉我从 C 到 B 的打车需要多久？”

高德 API 几百毫秒内返回数据，前端立刻更新卡片之间的“交通时间”提示。

最终确认： 用户拖拽调整满意后，点击“确认行程”按钮。前端将最终的数组顺序发回给后端的 Node.js 进行存档，或者再次发给 Coze 让它根据最终路线生成一份“分享文案”。

## 视觉与交互体验优化
占位符与动画： 在拖拽过程中，利用 CSS 给原本的位置留出一个“虚线框”（Ghost class），并增加平滑的过渡动画，让拖拽手感更“粘滞”、更高级。

交通耗时悬浮提示： 在两张卡片的间隙处，设计一个带有小汽车（Van）图标的模块。拖拽顺序改变后，这里数字（如：打车 15 分钟）要有动态跳变的数字效果。

# 高德API使用
在 Vue 3 中调用高德地图（AMap）API 进行路线绘制，最优雅的方式是使用官方提供的 `@amap/amap-jsapi-loader`。针对你提到的“拖拽式规划”，核心逻辑是：监听前端列表顺序的变化，提取坐标点数组，然后调用高德的路径规划插件进行重新渲染。

实现步骤
1. **安装加载器**：
   `npm i @amap/amap-jsapi-loader --save`

2. **初始化地图与插件**：
   在 Vue 组件中引入并初始化地图，同时加载 `AMap.Driving`（驾车）插件。

3. **调用路径规划服务**：
   使用插件的 `search` 方法，传入起始点、终点及途经点（Waypoints）。对于分段耗时，通过循环调用两点间的 `search` 来获取。

## 核心代码实现逻辑
1. **地图初始化**
```javascript
import AMapLoader from '@amap/amap-jsapi-loader';

let map = null;
let driving = null;

AMapLoader.load({
    key: "你的高德Key",
    version: "2.0",
    plugins: ['AMap.Driving']
}).then((AMap) => {
    map = new AMap.Map("map-container", { center: [116.397, 39.908], zoom: 12 });
    driving = new AMap.Driving({
        map: map,
        hideMarkers: true // 隐藏默认标记，由前端自定义渲染序号标记
    });
});
```

2. **根据拖拽后的列表重绘路线与计算耗时**
```javascript
const drawRoute = () => {
    if (!driving || itineraryList.value.length < 2) return;

    const start = [itineraryList.value[0].lng, itineraryList.value[0].lat]; 
    const end = [itineraryList.value[itineraryList.value.length - 1].lng, itineraryList.value[itineraryList.value.length - 1].lat];
    const waypoints = itineraryList.value.slice(1, -1).map(item => [item.lng, item.lat]);

    driving.search(start, end, { waypoints }, (status, result) => {
        if (status === 'complete') {
            // 路线绘制成功后，分段计算每两个点之间的时间
            calculateSegmentTimes();
        }
    });
};
```

## 3. 多模式路径绘制实现 (Web 端)
针对不同的 `travel_mode`，前端需要调用高德地图不同的插件进行路径规划和渲染。

### 3.1 核心插件加载
在初始化地图时，必须显式加载以下插件：
- `AMap.Driving`: 驾车路径规划
- `AMap.Transit`: 公共交通路径规划
- `AMap.Walking`: 步行路径规划
- `AMap.Cycling`: 骑行路径规划

### 3.2 样式区分规范
为了增强视觉辨识度，不同出行方式应使用不同的路径颜色：
- **驾车**: 蓝色 (`#409EFF`)
- **步行**: 绿色 (`#67C23A`)
- **公交**: 橙色 (`#E6A23C`)
- **骑行**: 灰色 (`#909399`)

### 3.3 实现逻辑说明
由于行程是分段的（例如：第一段驾车，第二段步行），前端需要：
1. **分段请求**：循环行程数组，根据当前点到下一点的 `travel_mode` 调用对应插件的 `search` 方法。
2. **手动绘线**：为了支持多颜色混合显示，不直接使用插件的默认渲染（`map: map`），而是获取规划结果中的 `path` 坐标数组，手动创建 `AMap.Polyline` 实例并设置对应颜色。
3. **清理机制**：在重排行程或更改模式时，需先清理地图上已有的所有 `Polyline` 覆盖物。

# AI 提示词 (Prompt) 模块
为了让 AI 能够稳定、准确地生成符合前端地图规划组件要求的数据格式，请在智能体（Bot）的 **人设与回复逻辑** 中加入以下结构化约束。

## 1. 触发场景
当用户表达以下意图时触发：
- “帮我规划一下去北京的路线”
- “这几个景点怎么走比较顺路？”
- “我想去故宫、长城和天安门，怎么安排？”

## 2. 数据输出规范
AI 在回复的最后，必须紧跟一个由 `[MAP_DATA]` 标签包裹的 JSON 结构。

### JSON 结构定义：
```json
[MAP_DATA]
{
  "itinerary": [
    {
      "id": "String, 唯一标识如 spot_01",
      "name": "String, 地点名称",
      "type": "String, 类型: departure(出发地)/attraction(景点)/restaurant(餐厅)/hotel(酒店)",
      "coordinates": {
        "lng": "Number, 经度 (GCJ-02坐标系)",
        "lat": "Number, 纬度 (GCJ-02坐标系)"
      },
      "suggested_duration": "Number, 建议停留时长(分钟)",
      "travel_mode": "String, 从此点到下一点的出行方式: driving/walking/transit/cycling",
      "description": "String, 地点简短介绍"
    }
  ]
}
[/MAP_DATA]
```

## 3. 提示词 (Prompt) 示例
请将以下内容配置到智能体的“提示词”或“工作流说明”中：

> ### 🗺️ 地图路线规划指令 (核心)
> 1. **强制性**：当你推荐超过 2 个地点时，必须在回复末尾附带 `[MAP_DATA]` JSON 块。
> 2. **必须包含出发地**：`itinerary` 数组的**第一个元素**必须是用户的起始位置，且 `type` 必须为 `departure`。
> 3. **出行方式规划**：为每一个地点指定 `travel_mode`（从该点到下一点的推荐方式）。可选值：`driving`(驾车)、`walking`(步行)、`transit`(公共交通)、`cycling`(骑行)。
> 4. **坐标系要求**：必须使用 **高德地图坐标系 (GCJ-02)**。
> 5. **格式纯净度**：`[MAP_DATA]` 标签内部只能包含合法的 JSON 代码。
> 6. **示例格式**：
> [MAP_DATA]{"itinerary":[{"id":"start_01","name":"洛阳龙门站","type":"departure","coordinates":{"lng":112.463,"lat":34.593},"suggested_duration":0,"travel_mode":"driving","description":"行程起始点"},{"id":"spot_01","name":"龙门石窟","type":"attraction","coordinates":{"lng":112.479,"lat":34.556},"suggested_duration":180,"travel_mode":"walking","description":"中国石刻艺术宝库"},{"id":"spot_02","name":"白马寺","type":"attraction","coordinates":{"lng":112.601,"lat":34.721},"suggested_duration":120,"travel_mode":"driving","description":"佛教传入中国后兴建的第一座官办寺院"}]}[/MAP_DATA]
