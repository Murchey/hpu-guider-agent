# 前端技术问答指南（答辩用）

> 本文档面向后端同学，帮助你在评委提问时能清晰阐述前端的技术选型、架构设计和开发思路。

---

## 一、技术栈总览

| 层级 | 技术 | 版本 | 选用理由 |
|------|------|------|----------|
| 框架 | Vue 3 (Composition API) | 3.4+ | 组合式 API 逻辑复用更灵活，TypeScript 友好 |
| 语言 | TypeScript | 5.x | 编译期类型检查，减少运行时错误 |
| 构建工具 | Vite | 5.x | 秒级启动、原生 ESM、构建速度快 |
| UI 组件库 | Element Plus | 2.13+ | Vue 3 生态最成熟的组件库，表单/弹窗/抽屉开箱即用 |
| 状态管理 | Pinia | 3.x | Vue 3 官方推荐，替代 Vuex，API 更简洁 |
| 地图 | 高德地图 JS API 2.0 | — | 国内地图服务首选，路径规划插件丰富 |
| Markdown | markdown-it | 14.x | 轻量高性能，支持插件扩展（linkify/breaks） |
| 富文本安全 | DOMPurify | 3.x | 防止 AI 输出的 HTML 中的 XSS 注入 |
| 拖拽排序 | vue-draggable-plus | 0.6+ | 基于 Sortable.js，Vue 3 响应式兼容 |
| HTTP 请求 | Axios + Fetch API | — | Axios 处理 REST 请求，Fetch 处理 SSE 流式 |
| 移动端打包 | Capacitor + Android SDK | 6.x | 将 Web 应用包装为原生 Android APK，无需编写 Java 代码 |
| 桌面端打包 | Electron | — | 将 Web 应用打包为 Windows 桌面程序，复用同一套前端代码 |

**评委可能问：为什么不用 React / Next.js？**
> 项目是单页应用（SPA），不需要 SSR。Vue 3 的组合式 API + TypeScript 在中等规模项目中开发效率更高，学习曲线更平缓，团队上手更快。

## 二、多端打包方案

项目采用 **一套代码、多端运行** 的策略，前端 Vue 3 代码只需构建一次，即可分别打包为 Android 移动应用和 Windows 桌面应用。

```
                    ┌─── vite build ───┐
                    │                  │
                    ▼                  ▼
            Web 产物 (dist/)     Web 产物 (dist/)
                    │                  │
         Capacitor sync          Electron loadFile
                    │                  │
                    ▼                  ▼
            Android APK         Windows .exe
```

### 2.1 移动端：Capacitor + Android SDK

**打包原理**：Capacitor 将 Vite 构建产物（`dist/`）嵌入一个 Android WebView 容器中，生成标准的 Android 工程，再通过 Android SDK 编译为 APK。

**核心流程**：
1. `vite build` 生成 Web 产物
2. `npx cap sync android` 将产物同步到 Android 工程的 `app/src/main/assets/public/`
3. Android Studio 或 `gradlew assembleDebug` 编译出 APK

**关键说明**：我们没有编写任何 Java/Kotlin 原生代码，完全依赖 Capacitor 的 WebView 壳 + 前端逻辑。这意味着所有业务逻辑（AI 对话、地图渲染、拖拽排序等）都在 Web 层完成，Android 端只负责提供容器。

**评委可能问：为什么不直接用原生 Android 开发？**
> 1. **开发效率**：一套 Vue 代码同时服务于 Web、Android、Electron 三端，不需要维护三套代码
> 2. **功能无差异**：我们的核心功能（SSE 流式输出、高德地图、拖拽排序）全部基于 Web API，原生 WebView 完全支持
> 3. **团队能力**：前端技术栈为主，Capacitor 让我们无需学习 Java/Kotlin 即可产出 Android 应用

**评委可能问：WebView 性能够用吗？**
> 够用。高德地图 JS API 本身就是在浏览器中运行的，Capacitor 的 WebView 底层使用的是 Android 系统的 Chromium 内核，性能与 Chrome 浏览器一致。我们还通过 `shallowRef`、防抖、分包等手段做了专项优化。

### 2.2 桌面端：Electron

**打包原理**：Electron 将 Vite 构建产物（`dist/`）加载到一个 Chromium 窗口中，通过 `electron-builder` 打包为 Windows 安装包（NSIS 安装程序）。

**核心代码**（`Sample/main.js`）：
```javascript
const { app, BrowserWindow } = require('electron')

function createWindow() {
  const win = new BrowserWindow({
    width: 1200,
    height: 800,
    webPreferences: {
      nodeIntegration: false,
      contextIsolation: true   // 安全隔离
    },
    autoHideMenuBar: true
  })
  win.loadFile(path.join(__dirname, 'dist/index.html'))
}

app.whenReady().then(createWindow)
```

**打包配置**（`Sample/package.json`）：
- 构建目标：`nsis`（Windows 安装程序），架构 `x64`
- 打包内容：`dist/**/*`（前端产物）+ `main.js`（Electron 入口）
- 输出目录：`release/`

**已解决的问题**：
- **构建产物同步**：Electron 的 `dist/` 需要与主项目 `vite build` 的产物保持同步，通过 `sync:dist` 脚本自动完成
- **流式输出兼容**：Electron 的 Chromium 内核对 SSE 的支持与浏览器一致，无需特殊处理
- **缓存清理**：调试阶段可启用 `disable-http-cache` 避免旧资源缓存

**评委可能问：为什么不直接做成网页就行，还要打包成桌面程序？**
> 1. **离线可用**：Electron 应用不依赖浏览器标签页，关闭后可在任务栏快速恢复
> 2. **更好的系统集成**：支持系统通知、文件访问、开机自启等原生能力
> 3. **用户体验**：独立窗口、独立图标，更像一个"正经软件"而非网页

## 三、项目结构

```
src/
├── main.ts                  # 入口：Pinia、Element Plus 初始化、主题加载
├── App.vue                  # 根组件：Tab 导航 + 暗色模式切换
├── env.d.ts                 # 环境变量类型声明
├── components/
│   ├── IndexPage.vue        # 首页：视差滚动、碎片飞散特效、用户画像问卷
│   ├── AiDialoguePage.vue   # AI 对话：流式输出、自定义标签协议、图片上传
│   ├── MapPlanningPage.vue  # 地图规划：多模式路径规划、拖拽排序、APP 唤起
│   └── SettingPage.vue      # 设置：API 配置热切换、System Prompt 管理
├── stores/
│   └── chatStore.ts         # Pinia store：对话会话持久化
└── assets/                  # 静态资源（图片、字体、图标）
```

**评委可能问：为什么没有用路由（vue-router）？**
> 项目只有 4 个页面且都在同一个 Tab 体系下，用 Element Plus 的 `el-tabs` 切换即可，引入 vue-router 会增加不必要的复杂度。页面间数据传递通过 props + emit 和 Pinia store 完成。

## 四、核心业务流程

```
用户填写画像问卷 → 存入 localStorage → 跳转对话页
                                           ↓
                              AI 接收画像，生成个性化欢迎语
                                           ↓
                        AI 输出 [SCENE_DATA] 标签 → 前端弹出单选卡
                                           ↓
                        用户选择 → 前端隐藏发送 [USER_CHOICE]
                                           ↓
                        AI 推荐景点 + [SELECT_DATA] → 前端弹出多选卡
                                           ↓
                        用户勾选 → 隐藏发送 [USER_CHOICE]
                                           ↓
                        AI 生成完整行程（含 [MAP_DATA] JSON）
                                           ↓
                        用户点击"转到地图" → 地图页解析 JSON 并绘制路线
```

**评委可能问：什么是"隐藏发送"？**
> 用户在弹窗中点击按钮后，前端会自动构造一条 `[USER_CHOICE]` 指令发送给 AI，但这条消息不会显示在聊天界面中。用户看到的是"点了一个按钮"，实际后端收到了一条结构化的指令，AI 据此推进到下一步。这保证了交互的简洁性。

## 五、AI 集成方案

### 5.1 流式输出（SSE）

采用 **Server-Sent Events** 协议实现实时打字机效果：

```
前端 fetch(POST /v3/chat, { stream: true })
    ↓
服务器持续推送 data: { "content": "你" }
                 data: { "content": "好" }
                 data: { "content": "呀" }
                 data: [DONE]
    ↓
前端逐行解析，拼接到页面
```

**关键技术点：**
- 使用 `ReadableStream` + `TextDecoder` 逐块读取
- 通过 buffer 机制处理网络波动导致的 JSON 截断
- 流结束后检查 buffer，兜底解析非流式 JSON 响应
- `AbortController` 支持用户中断请求，catch 中忽略 AbortError

**评委可能问：为什么不直接用 WebSocket？**
> SSE 是单向推送（服务端→客户端），正好符合 AI 生成文本的场景。相比 WebSocket，SSE 更简单、自动重连、基于 HTTP 协议无需额外握手。我们的场景不需要客户端频繁发消息，SSE 足够。

### 5.2 自定义交互协议

为了让 AI 不只是"输出文本"，还能触发前端 UI 交互，设计了基于标签的协议：

| 标签 | 作用 | 前端行为 |
|------|------|----------|
| `[MAP_DATA]{...}[/MAP_DATA]` | 携带地图数据 | 解析后显示"转到地图"按钮 |
| `[SCENE_DATA]{...}[/SCENE_DATA]` | 单选卡 | 弹出 1-3 个按钮供选择 |
| `[SELECT_DATA]{...}[/SELECT_DATA]` | 多选卡 | 弹出勾选列表 |
| `[USER_CHOICE]{...}[/USER_CHOICE]` | 用户反馈 | 隐藏发送给 AI |

**评委可能问：这个协议是你自己设计的吗？为什么不用 function calling？**
> 是的，自定义设计。因为我们的 AI 后端（Coze 智能体）原生支持文本输出，function calling 需要特定的模型支持且配置复杂。标签协议的优势是：纯文本实现、与任何 LLM 兼容、前端正则解析即可、AI 通过 Prompt 即可学会生成。实际上就是"Prompt Engineering 驱动的 UI 协议"。

### 5.3 多模态支持

- **Coze 模式**：图片通过 `/v1/files/upload` 接口上传，获取 `file_id` 随消息发送
- **Base64 模式**：图片转为 Base64 字符串直接嵌入请求体（兼容 OpenAI / 硅基流动）

用户可在设置页切换两种模式。

## 六、地图规划模块

### 6.1 整体方案

```
AI 返回 [MAP_DATA] JSON
    ↓
前端解析 itinerary 数组
    ↓
循环每两个相邻地点，根据 travel_mode 调用对应的高德插件
    ↓
driving → AMap.Driving
walking → AMap.Walking
transit → AMap.Transfer（优先）/ AMap.Transit（降级）
cycling → AMap.Cycling
    ↓
获取路径坐标，手动创建 Polyline（支持多颜色混合）
    ↓
地图自适应视野 setFitView()
```

### 6.2 跨城火车路线

高德的公交规划插件对跨城路线返回的火车段通常不包含铁轨坐标。解决方案：

1. 优先使用 `AMap.Transfer`（比 `AMap.Transit` 跨城支持更好）
2. 使用 Geocoder 动态获取城市名，调用 `setCity()` / `setCityd()`
3. 若仍返回 `NO_DATA`，降级到驾车规划，再降级到直线连接
4. 火车段缺少轨迹时，用 `getArcPath()` 生成正弦弧线模拟

**评委可能问：为什么用弧线而不是直线？**
> 直线连接两个城市在视觉上很生硬，不符合高铁/航空的视觉认知。正弦弧线模拟了大圆航线的效果，用户体验更自然。

### 6.3 样式规范

| 交通方式 | 颜色 | 线型 |
|----------|------|------|
| 驾车 | 鲜蓝 `#0084FF` | 实线 |
| 步行 | 青绿 `#13C2C2` | 虚线 |
| 公交/地铁 | 紫色 `#722ED1` | 实线 |
| 骑行 | 橙色 `#FA8C16` | 实线 |
| 火车 | 红色 `#F5222D` | 虚线弧线 |

## 七、状态管理

使用 **Pinia** 管理对话会话数据（`chatStore.ts`）：

```typescript
// 核心数据结构
interface ChatSession {
  id: string
  title: string          // 自动取第一条用户消息的前 20 字
  messages: Message[]    // role + content + imageUrl
  createdAt: number
}
```

**持久化策略**：通过 `watch` 监听 sessions 变化，自动序列化到 `localStorage`（键名 `ai-chat-sessions`）。页面刷新后自动恢复。

**支持的操作**：新建对话、切换对话、删除对话、重命名、修改消息（截断后续并重发）、清空全部。

**评委可能问：为什么不用后端存储对话？**
> 当前阶段优先保证前端体验完整性，localStorage 足以支撑单用户场景。后续可通过 Pinia store 的接口层无缝切换到后端 API 存储，业务逻辑无需改动。

## 八、性能优化

| 优化点 | 措施 |
|--------|------|
| 构建分包 | Vite 的 `manualChunks` 将 element-plus、vue 独立打包 |
| 地图实例 | 使用 Vue 3 的 `shallowRef` 避免 AMap 实例被 Proxy 劫持导致方法调用失败 |

| 内存释放 | 图片预览使用 `URL.revokeObjectURL()` 及时释放 |
| XSS 防护 | markdown-it 渲染结果经 DOMPurify 净化后再 v-html |

## 十、答辩高频问题速查

**Q：你们项目的创新点是什么？**
> 1. 自定义标签协议，让 AI 输出直接驱动前端 UI 交互（单选卡、多选卡、地图跳转），不只是纯文本对话
> 2. 多模式路径规划 + 火车弧线模拟，实现了从步行到跨城高铁的完整出行可视化
> 3. 拖拽式行程调整，前端实时调用高德 API 重算路线，所见即所得

**Q：你们怎么处理 AI 返回的数据不规范的问题？**
> 1. Prompt 工程：在 System Prompt 中给出严格的 JSON Schema 和示例（Few-shot）
> 2. 前端容错：正则匹配标签、JSON.parse 带 try-catch、解析失败时保留原始文本
> 3. 流式兜底：buffer 机制处理 JSON 截断，流结束后兜底检查非流式响应

**Q：地图路径规划失败了怎么办？**
> 三级降级策略：公交规划 → 驾车规划 → 起终点直线连接。确保地图上始终有可视化的路径，不会出现空白。

**Q：你们怎么做多轮对话的？**
> 两种方案：Coze 模式下持久化 user_id + 携带 additional_messages 历史；自定义模式下前端维护完整 messages 数组随请求发送。对话历史通过 Pinia store + localStorage 持久化，刷新不丢失。

**Q：前端和后端是怎么协作的？**
> 前端通过 Coze API 与 AI 后端通信（REST + SSE），后端负责 LLM 调用、Prompt 管理、知识库检索。前端专注于 UI 交互、地图渲染、流式展示。两端通过 `[MAP_DATA]` 等标签协议约定数据格式，松耦合。

**Q：项目部署方式？**
> 项目支持三端部署：Web 端通过 `vite build` 构建为纯静态文件，部署到 CDN 或静态服务器；Android 端通过 Capacitor 将 Web 产物嵌入 WebView，编译为 APK；桌面端通过 Electron 加载 `dist/index.html`，用 electron-builder 打包为 Windows 安装程序。三端共享同一套 Vue 3 前端代码。

**Q：Capacitor 和原生 Android 开发有什么区别？**
> Capacitor 本质上是一个 WebView 容器，前端代码以网页形式运行在其中。我们没有编写任何 Java/Kotlin 代码，所有业务逻辑都在 Web 层实现。优势是开发效率高、一套代码多端复用；劣势是无法使用深度原生能力（如蓝牙、NFC）。对于我们的场景（AI 对话 + 地图），WebView 完全够用。

**Q：Electron 打包后体积大吗？**
> Electron 需要内嵌一个 Chromium 引擎，安装包约 100-150MB。这是 Electron 的固有开销，换来的是跨平台一致性和开发效率。如果对体积敏感，可考虑 Tauri（使用系统 WebView），但需要 Rust 语言支持，对团队要求更高。

**Q：三个端的功能完全一致吗？**
> 是的。三端共享相同的 Vue 3 源码，通过 Vite 一次构建，产出相同的 `dist/` 产物。Capacitor 和 Electron 只是不同的"壳"，不修改任何前端逻辑。唯一差异是 Android 端的地图 APP 唤起功能更实用（可直接跳转高德地图 APP），Web 端则走 H5 版路线规划。

**Q：你们为什么没有适配鸿蒙版本？**
> 鸿蒙（HarmonyOS NEXT）与 Android 是两套完全独立的操作系统，无法通过一套代码同时适配，原因如下：
> 1. **内核不同**：Android 应用运行在 ART 虚拟机上，使用 Java/Kotlin；鸿蒙 NEXT 使用方舟运行时（ArkRuntime），开发语言是 ArkTS（基于 TypeScript 但有自己的 UI 框架 ArkUI）。两者不兼容
> 2. **没有 WebView**：这是最关键的区别。Android 有成熟的 WebView 组件，Capacitor 正是基于此将前端代码嵌入原生容器。而鸿蒙 NEXT 从底层去掉了 Android 的 WebView 机制，虽然有"Web 组件"但功能和兼容性远不及 Chromium WebView，无法承载我们复杂的地图渲染和 SSE 流式输出
> 3. **UI 框架不互通**：我们的前端基于 Vue 3 + DOM 操作，而鸿蒙的 ArkUI 采用声明式 UI 范式，组件体系完全不同。不存在类似 Capacitor 这样能把 Web 应用直接包装成鸿蒙应用的成熟工具
> 4. **地图 SDK 不同**：高德地图 JS API 是浏览器端的，鸿蒙需要使用高德的鸿蒙原生 SDK（如果已适配），接口和调用方式完全不同
>
> 简单来说：Capacitor 能工作是因为 Android 有 WebView，而鸿蒙没有等价的 WebView 生态。要适配鸿蒙就需要用 ArkTS 重写整个前端，相当于一个全新项目，这在当前阶段的人力和时间条件下不现实。未来如果鸿蒙生态成熟、出现类似 Capacitor 的跨端方案，可以考虑适配。
